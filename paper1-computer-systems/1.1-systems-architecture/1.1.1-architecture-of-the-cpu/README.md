## üìå Purpose of the CPU
The **Central Processing Unit (CPU)** is the brain of the computer. It is responsible for:
- Executing program instructions
- Performing arithmetic and logic operations
- Managing the flow of data within the system

---

## üîÑ The Fetch-Execute Cycle

## üß† Step-by-Step Breakdown

### 1. Fetch
- The **Program Counter (PC)** holds the address of the next instruction.
- This address is copied into the **Memory Address Register (MAR)**.
- The instruction is fetched from RAM and loaded into the **Memory Data Register (MDR)**.
- The **PC** is incremented to point to the next instruction.

### 2. Decode
- The **Control Unit (CU)** decodes the instruction in the MDR.
- The CU determines the required operation.

### 3. Execute
- The instruction is executed:
  - The **ALU** performs calculations or logic operations.
  - Data may be stored or retrieved from memory.
  - The result may be stored in the **Accumulator**.

# üèõÔ∏è Von Neumann Architecture: CPU Registers Explained (with Examples)

Von Neumann Architecture uses **registers** to temporarily hold **addresses** or **data** while instructions are being processed. These are essential to the **Fetch-Decode-Execute Cycle**.

---

## üìã Register Summary

| Register | Role |
|----------|------|
| **MAR** (Memory Address Register) | Stores the **address** of the next instruction or data to be fetched from memory |
| **MDR** (Memory Data Register) | Stores the **actual data** fetched from memory or to be written back |
| **PC** (Program Counter) | Stores the **address of the next instruction** to execute |
| **Accumulator** | Stores the **results of calculations** done by the ALU |

---

## üéØ Real-World Analogies for Each Register

### üìç **MAR (Memory Address Register)**  
> Think of MAR as a **postcode finder**.

| Example | Description |
|--------|-------------|
| üó∫Ô∏è **GPS Navigation** | You input a destination address in your sat nav. That‚Äôs the MAR‚Äîit holds where to go (memory address), but not the contents of that location. |
| üì¶ **Amazon Delivery Label** | The address on a parcel tells the driver where to deliver (memory location), but not what's inside the box (the data). |

---

### üì¶ **MDR (Memory Data Register)**  
> MDR is the **actual content** being delivered.

| Example | Description |
|--------|-------------|
| üì¨ **Parcel Contents** | After reaching the destination using the address (MAR), the MDR is the package you collect‚Äîit contains the **data**. |
| üì± **Downloading a Photo** | You tap a photo to load it. The system knows where to find it (MAR), but MDR holds the image data you actually view. |

---

### üß≠ **PC (Program Counter)**  
> PC keeps track of **where you are in the program**.

| Example | Description |
|--------|-------------|
| üìñ **Bookmark in a Book** | You‚Äôre reading a book and use a bookmark to save your place. The PC is that bookmark‚Äîit tells the CPU which instruction to read next. |
| üéÆ **Level Progress Tracker** | In a game, it remembers your current level or mission‚Äîjust like the PC tracks which instruction is next in line. |

---

### üßÆ **Accumulator**  
> Stores **results** of operations from the ALU.

| Example | Description |
|--------|-------------|
| üß† **Mental Maths Memory** | While doing sums in your head, you temporarily store answers like "5 + 3 = 8"‚Äîjust like the accumulator does. |
| üìù **Working Column in a Quiz** | When solving maths questions, you might jot down intermediate answers. The accumulator holds these until you're done. |

---

## üîÑ How It All Works Together

When the CPU processes instructions:

1. **PC** tells the CPU where to fetch from.
2. That address is sent to the **MAR**.
3. The data at that address is fetched into the **MDR**.
4. The **CU** decodes the instruction.
5. If it‚Äôs a calculation, the **ALU** works it out and stores the result in the **Accumulator**.
